# 酒库功能优化方案

本文基于当前工程代码与数据结构整理，目标是把“酒库”从一个简单清单设计成可支撑筛选、推荐、LLM 约束和后续维护的领域能力。

## 当前工程概览

当前项目是一个本地运行的调酒学习 App：

- 后端：FastAPI，主要入口在 `app/main.py`。
- 数据：所有菜谱都在 `data/` 下用 Markdown 存储，不依赖数据库。
- 菜谱加载：`app/knowledge.py` 解析 Markdown，生成内存中的 `Recipe` 索引。
- 特调生成：`app/llm.py` 把知识库、短期记忆和相关菜谱拼进 prompt，通过 OpenAI-compatible 接口流式回复。
- 前端：`frontend/index.html`、`frontend/js/app.js`、`frontend/css/style.css`，纯 HTML/CSS/JS，无构建步骤。
- 记忆：`app/memory.py` 把调酒师记忆和已保存特调的设计对话存在 Markdown 文件中。

这套架构的优点是轻、透明、容易改：菜谱即数据，刷新即可读取。但目前“材料”仍主要是展示文本，系统不知道家里有哪些酒，也不知道某个配方缺了什么。

## 现有不足

### 1. 材料没有规范化

菜谱中的材料以自然语言存在，例如：

- `琴酒 60 ml`
- `黑麦威士忌或波本 60 ml`
- `柠檬皮或橄榄 1 份`
- `Honey-agave syrup ...`

这些文本适合人读，但不适合机器判断：

- 同一材料可能有中英文、别名、品牌名、类别名。
- “波本或黑麦威士忌”实际是可选组，不是必须同时拥有。
- 装饰、冰块、杯具、工具和核心材料混在同一张配方表里。
- 柠檬、青柠、柠檬汁、柠檬皮在库存判断中应有不同粒度。

### 2. 经典鸡尾酒筛选能力较弱

前端目前支持：

- 类型筛选：全部 / 经典 / 特调 / 无酒精
- 搜索：名称、基酒、杯型、风味、标签

但还不能回答用户真正关心的问题：

- 我现在能调哪些？
- 哪些缺一两样但可以合理替代？
- 哪些缺关键材料，暂时不建议做？

### 3. 调酒师没有库存约束

LLM prompt 里有菜谱知识、情绪、记忆和相关参考，但没有“当前吧台库存”。所以调酒师可能给出用户没有的材料，例如某种利口酒、苦精、糖浆或新鲜水果。

这会破坏“私人吧台”的体验：用户希望它像一个真的调酒师，先看眼前有什么，再设计能落地的一杯。

### 4. 替代判断不应完全实时依赖 LLM

“缺少金巴利能否替代 Negroni？”这类问题既需要领域知识，也需要一致性。如果每次筛选都实时问 LLM，会有这些问题：

- 慢，列表筛选体验差。
- 成本高。
- 结果不稳定，同一库存状态可能得到不同判断。
- 难解释：用户看不到为什么被归为“可替代可调”。

因此替代关系应当有本地规则和缓存，LLM 更适合用于离线补全、维护解释和处理长尾材料。

## 建议目标

新增“酒库”能力后，系统至少支持两条主线：

1. 菜谱页筛选经典鸡尾酒的可调状态：
   - `available`：当前材料齐全，可以调。
   - `substitutable`：缺少非关键材料，或存在已确认替代方案，可以调但风味会变化。
   - `missing`：缺少关键材料，不建议调制。

2. 调酒师基于当前酒库设计特调：
   - 优先使用库存中已有材料。
   - 缺材料时先询问或给出替代，而不是直接开一个不可执行配方。
   - 最终 JSON 配方中尽量只包含可用材料，或明确标注替代项。

## 数据模型建议

建议继续保持“数据即文件”的项目风格，在 `data/` 下新增酒库数据，而不是一开始引入数据库。

### 1. 材料规范表

新增 `data/cellar/ingredients.yml`。

用于描述所有可识别材料，包括基酒、利口酒、强化酒、苦精、糖浆、水果、香草、气泡水、盐、糖等。杯子不放入酒库，工具也不放入酒库。

示例：

```yaml
ingredients:
  - id: gin
    zh: 琴酒
    en: Gin
    category: base_spirit
    aliases: [金酒, London Dry Gin, 干金酒]
    essential_level: core

  - id: campari
    zh: 金巴利
    en: Campari
    category: bitter_liqueur
    aliases: [Campari]
    essential_level: core

  - id: lemon
    zh: 柠檬
    en: Lemon
    category: citrus
    aliases: [黄柠檬]

  - id: lemon_juice
    zh: 新鲜柠檬汁
    en: Fresh lemon juice
    category: citrus_juice
    aliases: [柠檬汁]
    parent: lemon

  - id: lemon_peel
    zh: 柠檬皮
    en: Lemon peel
    category: garnish
    parent: lemon
```

关键点：

- 用 `id` 做稳定机器标识。
- `aliases` 解决中英文、俗称、品牌名。
- `category` 支持同类替代。
- `parent` 处理“有柠檬即可得到柠檬汁/柠檬皮”的关系。
- `essential_level` 可作为默认关键性，但最终关键性应以配方里的角色为准。

### 2. 用户当前库存

新增 `data/cellar/inventory.yml`。

示例：

```yaml
inventory:
  - ingredient_id: gin
    status: in_stock
    note: London Dry

  - ingredient_id: sweet_vermouth
    status: in_stock

  - ingredient_id: campari
    status: missing

  - ingredient_id: lemon
    status: in_stock

  - ingredient_id: soda_water
    status: in_stock
```

状态建议：

- `in_stock`：有。
- `low`：快没了，仍可用于判断可调。
- `missing`：没有。
- `ignored`：不纳入判断，例如用户不关心的装饰。

后续可增加品牌、容量、开瓶日期、过期提醒，但第一版不必做复杂。

### 3. 替代规则与缓存

新增 `data/cellar/substitutions.yml`。

示例：

```yaml
substitutions:
  - missing: bourbon
    substitute: rye_whiskey
    confidence: high
    impact: 黑麦更干、更辛辣，整体仍成立。
    suitable_for: [old-fashioned, manhattan]

  - missing: campari
    substitute: aperol
    confidence: medium
    impact: 苦度下降、甜橙感更明显，不再是标准 Negroni，但可做轻盈变体。
    suitable_for: [negroni]

  - missing: fresh_lime_juice
    substitute: fresh_lemon_juice
    confidence: low
    impact: 酸型和香气会明显不同，酸酒结构仍可成立，但不建议标为经典原版。
```

替代建议分三层：

- 本地硬规则：常见、稳定、可解释的替代。
- LLM 维护缓存：后台批量分析缺失项，写入缓存文件，供前端快速使用。
- 实时 LLM：只用于用户追问“为什么这样替代”或长尾材料。

## 配方材料解析建议

当前 `Recipe.ingredients` 只有 `amount` 和 `item`。建议新增一个内部结构，不一定立刻暴露给 Markdown 作者：

```python
class NormalizedIngredient(BaseModel):
    ingredient_id: str
    amount: str = ""
    raw_item: str
    role: str = "ingredient"
    required: bool = True
    alternatives: list[str] = []
```

`role` 可选：

- `base`
- `modifier`
- `acid`
- `sweetener`
- `bitter`
- `dilution`
- `carbonation`
- `garnish`
- `ice`

判断原则：

- 杯子不进入酒库。
- 冰块默认视作常备，不影响可调状态，除非未来用户明确关闭。
- 装饰默认不作为关键材料，但柑橘皮、盐边、薄荷这类会明显影响香气的装饰，可标为 `required: false` 但影响替代说明。
- “A 或 B”要解析为 alternatives，而不是两个必需项。

## 后端能力设计

建议新增 `app/cellar.py`，负责材料规范、库存、替代、可调性分析。

### 主要函数

```python
def load_ingredients() -> IngredientCatalog
def load_inventory() -> Inventory
def normalize_recipe_ingredients(recipe: Recipe) -> list[NormalizedIngredient]
def evaluate_recipe_availability(recipe: Recipe, inventory: Inventory) -> AvailabilityResult
def build_cellar_context() -> str
```

### AvailabilityResult

```python
class AvailabilityResult(BaseModel):
    status: str  # available | substitutable | missing | unknown
    available: list[str] = []
    missing: list[str] = []
    substitutable: list[SubstitutionHit] = []
    non_blocking_missing: list[str] = []
    explanation: str = ""
```

### API 建议

第一版可加入：

- `GET /api/cellar`：返回材料目录、当前库存、统计。
- `PATCH /api/cellar/inventory`：更新某个材料的库存状态。
- `GET /api/cocktails?availability=available|substitutable|missing`：菜谱列表按可调状态过滤。
- `GET /api/cocktails/{slug}/availability`：查看某一杯为什么可调或不可调。

后续再考虑：

- `POST /api/cellar/refresh-substitutions`：用 LLM 批量刷新替代缓存。
- `GET /api/cellar/suggestions`：根据现有菜谱反推“买哪瓶酒解锁最多经典酒”。

## 前端交互建议

### 菜谱页

在现有类型 chip 下新增一组“可调状态”筛选：

- 全部
- 现在可调
- 可替代可调
- 缺关键材料

每张卡片显示一个小状态：

- `可调`
- `可替代`
- `缺：金巴利、甜味美思`

打开详情页时，在配方上方显示“库存判断”：

- 已有材料打勾。
- 缺失关键材料标红。
- 可替代项展示替代建议和风味影响。

### 设置页或新增酒库页

第一版可以放在设置页里，作为“我的酒库”区块：

- 按分类展示：基酒、利口酒/开胃酒、强化酒、苦精、糖浆、果汁/水果、气泡/软饮、香草/装饰。
- 每项一个开关或分段按钮：有 / 没有 / 忽略。
- 支持搜索材料。

如果后续功能变多，再把“酒库”提升为顶部第三个主 tab。

## LLM prompt 集成

在 `KnowledgeBase.build_chat_context()` 或新建上下文拼接层中加入：

```markdown
## 当前酒库 / Current cellar
用户当前拥有：
- Gin 琴酒
- Sweet vermouth 甜味美思
- Lemon 柠檬
- Soda water 苏打水

当前缺少：
- Campari 金巴利

## 设计约束
设计特调时，优先使用“当前拥有”的材料。若需要缺失材料，必须先说明并询问是否接受替代；最终配方应默认只使用已有材料或已确认替代材料。
```

同时修改 `SYSTEM_PROMPT` 中“当要给出完整配方时”的规则：

- 最终 JSON 的 `ingredients` 默认必须来自当前酒库。
- 如果使用替代项，建议在 `bartender_notes` 中写清楚“原本会用 X，现在用 Y，风味影响是 Z”。
- 不要把杯具作为库存限制。

## LLM 替代缓存策略

替代判断建议后台维护，不在列表筛选时实时调用 LLM。

流程：

1. 本地规则先跑一遍，得到已知替代。
2. 找出未知缺失项，例如“缺 Elderflower liqueur 是否能替代？”。
3. 后台 LLM 生成结构化建议。
4. 人工或自动写入 `data/cellar/substitutions.cache.yml`。
5. 前端筛选只读本地缓存，保证速度和稳定性。

缓存条目建议包含：

```yaml
- recipe_slug: negroni
  missing: campari
  substitute: aperol
  verdict: substitutable
  confidence: medium
  explanation: 苦度和药草感下降，但可做轻盈变体。
  updated_at: "2026-07-07"
```

## 分阶段落地建议

### Phase 1：本地酒库和可调筛选

- 新增 `data/cellar/ingredients.yml` 和 `inventory.yml`。
- 新增 `app/cellar.py`。
- 先覆盖现有经典鸡尾酒中出现的常见材料。
- 后端列表返回 `availability` 字段。
- 前端增加“现在可调 / 缺关键材料”筛选。

这一阶段不做 LLM 替代，只判断“齐全/缺失”。

### Phase 2：替代规则

- 新增 `substitutions.yml`。
- 支持 `substitutable` 状态。
- 详情页展示替代说明。
- 给常见经典鸡尾酒补充人工确认替代规则。

### Phase 3：调酒师库存约束

- 把当前库存拼进 chat context。
- 更新 prompt，要求最终配方优先使用库存。
- 保存特调时也记录当时使用的材料状态，方便以后复盘。

### Phase 4：LLM 后台刷新缓存

- 增加“刷新替代建议”接口。
- 将 LLM 输出限制为结构化 JSON/YAML。
- 对低置信度建议标记为待确认，不直接用于“可替代可调”。

### Phase 5：购买建议

当酒库稳定后，可以做更有价值的功能：

- “买一瓶金巴利可以解锁 Negroni、Americano、Boulevardier。”
- “你现在最缺的是甜味美思，它能解锁 N 杯经典。”
- “按预算推荐下一瓶酒。”

## 实现风险

- 材料规范化会遇到很多边界词：品牌、类别、风味描述、装饰、工具容易混淆。
- 替代关系不能只有类别判断，比如所有苦味利口酒都不是 Campari 的等价替代。
- 柑橘类要区分水果、果汁、皮、角、轮片。
- 用户库存也有粒度问题：“有威士忌”不等于“有黑麦威士忌”。
- LLM 生成配方可能仍会越界，需要 prompt、后端校验和前端提示共同约束。

## 推荐第一步

建议先做一个很小但闭环的版本：

1. 只覆盖当前经典鸡尾酒文件中出现的材料。
2. 不处理品牌和容量。
3. 冰块默认有，杯具不纳入酒库。
4. 装饰不阻塞可调，但展示为“缺少非关键材料”。
5. 先实现 `available` 和 `missing` 两类。
6. 等本地判断稳定后，再加入 `substitutable`。

这样可以快速让用户在经典鸡尾酒列表中看到“我现在能调什么”，同时为调酒师库存约束打下数据基础。
